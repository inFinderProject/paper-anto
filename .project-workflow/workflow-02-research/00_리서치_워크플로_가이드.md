# Workflow 02: 리서치 워크플로 가이드

> 목적: `ppt-intake-app`에서 인풋을 받은 이후, ChatGPT Desktop(에이전트 모드, 딥리서치)을 활용해 **콘텐츠 작성에 부족함이 없을 정도로 리서치를 풍부하게 진행**하기 위한 구조화된 플로우와 프롬프트 템플릿을 정의합니다.

## 1. 전체 흐름 (인풋 이후)

1. `ppt-intake-app`에서 프로젝트 정보를 제출해 브리프/요약 생성
2. 생성된 브리프(특히 AI 협업용 요약 + 프로젝트 메타데이터)를 ChatGPT Desktop 새 대화에 붙여넣기
3. 아래 **리서치 항목 리스트(R1~R6)** 중 필요한 것만 선택
4. 각 항목에 대응하는 프롬프트 템플릿을 사용해  
   - 에이전트 모드 또는  
   - 딥리서치 모드  
   로 리서치 실행
5. 리서치 결과를 프로젝트 노트/브리프에 정리 후, 콘텐츠·레이아웃 단계로 이동

공통 변수(모든 프롬프트에 사용):
- `{PROJECT_NAME}`: 프로젝트명
- `{CLIENT_NAME}`: 클라이언트명
- `{PROJECT_TYPE}`: 문서 유형(수주 제안서/내부 보고/IR/마케팅 등)
- `{INDUSTRY}`: 산업/도메인
- `{TARGET_AUDIENCE}`: 타겟(의사결정자/고객 등)
- `{GOAL}`: 이번 문서의 핵심 목표
- `{BRIEF_SUMMARY}`: ppt-intake-app에서 생성된 AI 협업용 요약 또는 핵심 요약

## 2. 리서치 항목 리스트 (R1~R6)

- **R1. 도메인·시장 리서치**  
  - 업계 트렌드, 시장 규모, 핵심 이슈, 주요 키워드 수집
- **R2. 클라이언트 & 브랜드 리서치**  
  - 클라이언트의 사업 구조, 포지셔닝, 강·약점, 공식 메시지 파악
- **R3. 타깃 & 이해관계자 리서치**  
  - 의사결정자/최종 사용자 특성, 니즈, 저항 포인트, 의사결정 구조
- **R4. 경쟁사 & 벤치마크 리서치**  
  - 직접 경쟁사/유사 서비스, 레퍼런스 사례, 차별화 포인트 정리
- **R5. 증거·데이터·인사이트 리서치**  
  - 통계, 리포트, 인용 가능한 문장, 사례 데이터 등 “설득력 있는 근거”
- **R6. 스토리라인 & 슬라이드 구조 레퍼런스**  
  - 유사 목적의 덱 구조 패턴, 섹션 구성 아이디어, 논리 흐름 패턴

※ 모든 항목을 매번 실행할 필요는 없고, 프로젝트 난이도/시간에 따라 선택적으로 사용합니다.

## 3. 리서치 범위 및 깊이 정책

> 목표: “많이 아는 것”이 아니라 **이번 제안의 설득을 위해 필요한 만큼만, 빠르게** 리서치하는 것을 원칙으로 합니다.

### 3.1 기본 원칙
- 리서치 질문은 항상 **이번 제안서/기획서의 목표와 목차 섹션**에 연결해서 정의한다.
- 각 R코드(R1~R6)는 “이 프로젝트에서 진짜 필요한 것만” 선택하고, 나머지는 과감히 생략한다.
- Deep Research는 **핵심 1~2개 항목**(예: R1 시장, R5 증거)에만 사용하고, 나머지는 Agent 모드의 Quick 리서치로 처리한다.

### 3.2 레벨 정의 (프로젝트 시작 시 선택)
- **L0 Skip**: 이번 프로젝트에서는 건드리지 않음.  
  - 예: 이미 충분히 알고 있는 산업/클라이언트.
- **L1 Quick (기본)**: Agent 모드로 10~15분 이내, 3~5개의 인사이트/데이터만 수집.  
  - 예: R2 클라이언트, R3 타깃, R4 경쟁사의 개요 수준.
- **L2 Standard**: Quick + 필요한 경우 Deep Research 1회, 20~40분 이내.  
  - 예: R1 시장, R5 증거·데이터.
- **L3 Deep (예외)**: 연구형 문서나 매우 중요한 제안에서만 사용, 명시적으로 결정했을 때만 진행.

실무에서는 각 프로젝트마다 R1~R6에 대해 **L0/L1/L2 중 하나만 체크**하고, L3는 드물게 사용하는 것을 권장합니다.

### 3.3 종료 기준
- 각 주요 섹션(문제/기회/해결/효과)에 대해:
  - 설명용 인사이트 또는 예시가 **3~5개 이상 모였고**
  - 2차 목차를 수정·보완할 논리가 충분히 생겼다면  
  → 해당 R코드 리서치는 종료합니다.
- 같은 내용/출처가 반복되기 시작하면 “수확 체감 구간”으로 보고 멈춥니다.

## 4. ChatGPT Desktop 사용 방법 (요약)

1. ChatGPT Desktop에서 새 대화 시작 → 프로젝트명으로 북마크
2. 인트로 메시지에 `{BRIEF_SUMMARY}`와 주요 메타데이터를 붙여넣기
3. 아래 템플릿에서 **원하는 R코드(R1~R6)**를 선택해 프롬프트 복사
4. 중괄호 변수들을 실제 값으로 채운 뒤 실행  
   - 빠른 탐색/정리: **에이전트 모드**  
   - 깊이 있는 조사/자료 수집: **딥리서치 모드**
5. 결과를 프로젝트 노트에 섹션별로 정리(예: “R1 시장 리서치 요약” 등)

## 5. 리서치 항목별 프롬프트 템플릿

아래 템플릿은 모두 ChatGPT Desktop에서 사용하며,  
에이전트 모드와 딥리서치 모드를 구분해 제공합니다.

### R1. 도메인·시장 리서치

**[Agent Mode] R1_도메인_시장_개요**
```markdown
당신은 B2B/B2C 제안서·기획서 작성을 지원하는 리서치 컨설턴트입니다.

아래 프로젝트 정보를 바탕으로, {INDUSTRY} 도메인에 대한
시장·트렌드 개요를 정리해 주세요.

- 프로젝트명: {PROJECT_NAME}
- 클라이언트: {CLIENT_NAME}
- 문서 유형: {PROJECT_TYPE}
- 산업/도메인: {INDUSTRY}
- 타겟: {TARGET_AUDIENCE}
- 목표: {GOAL}
- 브리프 요약:
{BRIEF_SUMMARY}

요청:
1. 현재 {INDUSTRY}에서 중요한 트렌드 5~7개를 한 줄 요약과 함께 정리
2. 시장 규모/성장 방향을 설명할 때 유용한 관점 3~5개 제안
3. 제안서에 직접 쓸 수 있는 인사이트형 문장 5~7개 작성

출력 형식:
- 섹션1: 트렌드 리스트
- 섹션2: 시장 설명 관점
- 섹션3: 인사이트 문장 모음
```

**[Deep Research] R1_시장_깊이_조사**
```markdown
딥리서치 모드로, {INDUSTRY} 시장에 대해
제안서/기획서 작성에 바로 활용할 수 있는 수준의
깊이 있는 리서치를 수행해 주세요.

입력 정보:
- 프로젝트명: {PROJECT_NAME}
- 클라이언트: {CLIENT_NAME}
- 문서 유형: {PROJECT_TYPE}
- 산업/도메인: {INDUSTRY}
- 목표: {GOAL}

요청:
1. 최근 3~5년간의 주요 변화와 앞으로 3년 전망 요약
2. 대표적인 세분 시장/세그먼트 구조 제안
3. 자주 인용되는 리포트/조사기관/지표를 정리(출처 간단 표기)
4. 제안서 본문에서 사용할 수 있는 수치·사례 10개 이상 수집

출력 형식:
- 요약(3~5문단)
- 세그먼트 구조(불릿 또는 표)
- 지표/출처 리스트
- 인용용 문장/데이터 모음
```

### R2. 클라이언트 & 브랜드 리서치

**[Agent Mode] R2_클라이언트_브랜드_개요**
```markdown
당신은 제안서용 클라이언트 분석을 돕는 컨설턴트입니다.

프로젝트 정보를 바탕으로 {CLIENT_NAME}의 사업/브랜드 관점을 정리해 주세요.
웹에 공개된 일반 정보를 기반으로 해석하되, 내부 기밀은 다루지 마세요.

- 프로젝트명: {PROJECT_NAME}
- 클라이언트: {CLIENT_NAME}
- 산업/도메인: {INDUSTRY}
- 목표: {GOAL}

요청:
1. {CLIENT_NAME}의 사업 개요와 핵심 제품/서비스 정리
2. 시장에서의 포지셔닝(강점/약점/기회/위협 요약)
3. 제안서 톤앤매너를 잡을 때 참고할 만한 키워드 10개 제안
```

**[Deep Research] R2_클라이언트_깊이_조사**
```markdown
딥리서치 모드로 {CLIENT_NAME}에 대한 공개 정보를 조사해 주세요.

요청:
1. 최근 3년간 주요 이슈/뉴스/전략 변화 타임라인
2. 공식 웹사이트/IR/보도자료 기준 주요 메시지와 키워드
3. 시장/투자자/고객 관점에서 언급되는 평가 포인트(긍정/우려 각각)
4. 이번 프로젝트의 제안서/기획서에서 강조해야 할 방향성 제안 5가지
```

### R3. 타깃 & 이해관계자 리서치

**[Agent Mode] R3_타깃_페르소나**
```markdown
아래 정보를 기반으로, 이번 문서의 핵심 타깃(의사결정자/사용자)에 대한
페르소나와 니즈를 정리해 주세요.

- 타깃: {TARGET_AUDIENCE}
- 문서 유형: {PROJECT_TYPE}
- 목표: {GOAL}
- 브리프 요약:
{BRIEF_SUMMARY}

요청:
1. 대표 페르소나 2~3개 정의(역할, 책임, KPI, 고민)
2. 각 페르소나가 이 문서에서 알고 싶어하는 5가지 질문 정리
3. 설득 포인트/저항 포인트를 각각 5개씩 정리
```

### R4. 경쟁사 & 벤치마크 리서치

**[Agent Mode] R4_경쟁사_레퍼런스_개요**
```markdown
{INDUSTRY}에서 {CLIENT_NAME}와 경쟁하거나 비교될 만한
경쟁사/레퍼런스 사례를 정리해 주세요.

요청:
1. 대표 경쟁사/대체 솔루션 3~5개와 한 줄 설명
2. 각 경쟁사 강점/약점 요약(제안서 관점에서 유용한 요소 위주)
3. 벤치마크할 만한 캠페인/프로젝트/제품 사례 3~5개 정리
```

**[Deep Research] R4_벤치마크_깊이_조사**
```markdown
딥리서치 모드로, 유사 목적을 가진 프로젝트/캠페인/제품 사례를
국내외에서 조사해 주세요.

요청:
1. 유사 사례 5~10개 리스트(간단 설명 + 링크/출처)
2. 각 사례에서 배울 수 있는 핵심 포인트 3개씩 정리
3. 이번 프로젝트에 적용 가능한 아이디어 10개 제안
```

### R5. 증거·데이터·인사이트 리서치

**[Deep Research] R5_증거_데이터_수집**
```markdown
제안서/보고서에 신뢰도를 높이는 "증거·데이터·인용"을 모으는 목적의
깊이 있는 리서치를 수행해 주세요.

- 산업/도메인: {INDUSTRY}
- 목표: {GOAL}
- 타깃: {TARGET_AUDIENCE}

요청:
1. 관련 통계/리포트/조사 결과에서 인용할 만한 문장 15~20개 수집
2. 각 인용문에 대해:
   - 출처(기관/연도/리포트명)
   - 문맥(어떤 맥락에서 나온 말인지)
3. "문제 심각성", "기회 규모", "해결 필요성"을 설명할 때
   바로 쓸 수 있는 문장 템플릿 10개 작성
```

### R6. 스토리라인 & 슬라이드 구조 리서치

**[Agent Mode] R6_슬라이드_구조_아이디어**
```markdown
아래 정보를 기반으로, 이번 {PROJECT_TYPE} 문서의
스토리라인/슬라이드 구조 아이디어를 제안해 주세요.

- 프로젝트명: {PROJECT_NAME}
- 목표: {GOAL}
- 타깃: {TARGET_AUDIENCE}
- 브리프 요약:
{BRIEF_SUMMARY}

요청:
1. 전체 슬라이드를 4~6개의 큰 섹션으로 나누어 제안
2. 각 섹션별 핵심 메시지와 포함해야 할 내용 불릿 5개 이내로 정리
3. layout-patterns의 패턴 카테고리를 참고해,
   각 섹션에 어울리는 레이아웃 유형을 추천(비교형/나열형/강조형 등)
```

---

이 문서는 **리서치 단계의 기본 가이드**입니다.  
실제 프로젝트에 맞게 R코드 선택, 변수 값, 요청 강도를 조정하면서  
자신만의 리서치 플레이북으로 계속 업데이트해 나가면 됩니다.

## 6. 03 이후 마이크로 리서치 재순환 규칙

> 2차 목차(03)가 결정된 이후에는, 구조를 크게 바꾸지 않는 선에서 **슬라이드/섹션 단위의 작은 리서치**만 추가로 수행합니다.

- 추가 리서치가 필요하면 프로젝트 폴더의 `02-research/` 아래에  
  `micro-research.md` 또는 섹션별 노트 파일을 만들어, 다음 형식으로 기록합니다.

  - 예: `[섹션 3-2] R5 / L1 – 병원 F&B 이용률 최신 수치 2~3개`

- 실행 시에는 이 가이드의 R코드별 프롬프트를 재사용하되,
  - 질문 범위를 “해당 슬라이드에서 사용할 내용”으로 좁히고
  - 기본적으로 **L1 Quick + Agent 모드**만 사용합니다.

- 마이크로 리서치는
  - 03에서 확정한 목차를 변경하기 위한 것이 아니라,
  - 이미 정해진 메시지를 더 설득력 있게 만들기 위한 **보강 작업**이라는 점을 유지합니다.

## 7. PDF 리포트 & Claude/Droid Skill 활용 규칙

> ChatGPT Agent/Deep Research가 PDF 리포트를 생성하는 경우, **PDF는 원본 아티팩트로 유지**하면서, 필요한 경우에만 Skill로 요약 MD를 만듭니다.

- 저장 위치와 이름
  - 프로젝트별로 `projects/<project-name>/02-research/assets/` 폴더에 PDF를 저장합니다.
  - 파일명은 `R1-<short-topic>.pdf` 형식(예: `R1-hospital-fnb-strategy.pdf`)으로, R코드와 용도를 포함해 짓습니다.

- Claude/Droid Skill 활용 기준
  - PDF 내용이 여러 섹션/슬라이드에 반복 활용될 정도로 중요하거나,
  - 나중에 검색·재사용이 필요하다고 판단되면  
    → **Claude Skill 또는 Droid Skill**로 텍스트를 추출·요약해 MD 파일을 생성합니다.

- MD 요약 파일 규칙
  - 위치: 해당 프로젝트의 `02-research/` 폴더 (예: `R1-hospital-fnb-strategy.md`).
  - 파일명: PDF와 동일한 토픽을 사용하는 `R1-<short-topic>.md` 형식.
  - 상단에 YAML frontmatter 권장:
    ```yaml
    ---
    type: research-note
    category: <도메인/R코드>
    status: done
    source: <원본 PDF 파일명 또는 경로>
    tags:
      - client/<클라이언트명>
      - research/R1
    ---
    ```
  - 본문 내용 권장 구조:
    1. 메타 정보(출처 PDF, 생성일, 사용한 Skill/프롬프트 짧게)
    2. 이 프로젝트에 직접 쓸 핵심 인사이트 5~10개
    3. 제안서에 넣을 인용/숫자만 정리(필요 시 페이지/섹션 참조)
  - 모든 PDF를 MD로 변환할 필요는 없으며, **“이 프로젝트에서 실제로 쓸 것만 요약”**한다는 기준을 유지합니다.
